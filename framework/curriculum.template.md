## Module activation protocol

When a module transitions from Planned to Active:

1. Create `modules/0N-<slug>.md` (copy `../../framework/module.template.md`).
2. Lock the BS-check questions **before** starting work.
3. Update the instance README's status table.
4. Update the artifact spec if the module modifies a tracked artifact.
5. Open a monthly retrospective stub.

---

## Reordering criteria (when to deviate)

Reorder a module if:

- A live opportunity demands a specific skill earlier (interview funnel, contracting opportunity).
- An artifact lands faster than planned and frees bandwidth.
- Real constraints (life, work) eat bandwidth; collapse or defer accordingly.
- Actual usage of an existing artifact surfaces a gap the current order doesn't address.

Do **not** reorder because a new paper / framework / tool feels exciting. That's the drift trap. See `../methodology.md#anti-drift-disciplines`.
